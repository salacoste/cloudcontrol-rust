## Successes

1. **Pattern reuse from device heartbeats** — Story 9.2 successfully replicated the existing device heartbeat pattern (DashMap + tokio::spawn timeout checker) with provider-specific tuning (60s timeout, 5s poll interval).

2. **Epic 8 lessons applied immediately** — All H1/M1/M2/M3 fixes from Epic 8 code reviews were proactively applied:
   - `body: String` for jQuery PUT endpoints
   - `{% raw %}` for Tera/Vue templates
   - `rows_affected()` for 404 detection
   - `tracing::warn!` on database errors
   - Input trimming

3. **N+1 query optimization** — Story 9.2 code review eliminated redundant `count_devices_by_provider()` calls by using `devices.len() as i64`, saving 1 query per provider.

4. **Heartbeat optimization** — Story 9.2 H1 fix restructured heartbeat to only call `update_provider_presence(id, true)` for NEW sessions. Existing sessions just reset timer, preserving `presenceChangedAt` for correct uptime display.

5. **Zero compilation errors on first attempt** — Both stories compiled cleanly on first implementation due to thorough pattern following.

## Challenges

1. **Heartbeat state management** — Story 9.2 required careful thinking about when to update `presenceChangedAt`. The fix: only update on state change (offline→online), not on every heartbeat.

2. **CamelCase serialization** — Story 9.1 code review caught that the template expects `presenceChangedAt` but Rust uses `presence_changed_at`. Required `#[serde(rename = "presenceChangedAt")]`.

3. **API response format consistency** — All provider endpoints needed to return `ProviderWithDevices` with the same structure, including `device_count` and `devices` array.

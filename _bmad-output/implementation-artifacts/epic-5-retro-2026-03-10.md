## Architecture Evolution

Each story layered API capabilities on the existing control infrastructure:

```
5-1: REST API            ->  /api/v1/ namespace with typed request/response models
5-2: WS Screenshot API   ->  /api/v1/devices/{udid}/screenshot/ws WebSocket endpoint
5-3: Status & Health     ->  /api/v1/status, /api/v1/health, /api/v1/metrics + MetricsTracker
5-4: CI/CD Examples      ->  OpenAPI 3.0 spec at /api/v1/openapi.json
5-5: JSON-RPC WebSocket  ->  /api/v1/ws/nio with JSON-RPC 2.0 protocol for device-agnostic automation
```

## Successes

1. **Strongly typed API layer** -- `api_response.rs` defines typed structs for all request/response payloads (`TapRequest`, `SwipeRequest`, `InputRequest`, `KeyEventRequest`, `BatchTapRequest`, etc.) with serde validation. Invalid payloads fail at deserialization before reaching business logic.
2. **Consistent error codes** -- All error responses follow a uniform structure: `{status: "error", error: "ERR_DEVICE_NOT_FOUND", message: "..."}`. Error codes use a naming convention (`ERR_` prefix + domain + cause) that enables programmatic error handling by CI/CD clients.
3. **MetricsTracker for observability** -- `MetricsTracker` in `state.rs` tracks screenshot latency (last 1000 samples with percentile calculation), WebSocket connection count (AtomicU32), and exposes Prometheus-compatible text format at `/api/v1/metrics`. The percentile computation sorts and indexes without external crate dependencies.
4. **Health check with subsystem validation** -- `/api/v1/health` checks database connectivity and connection pool state. Returns HTTP 200 when healthy, HTTP 503 when unhealthy. This enables Kubernetes liveness/readiness probes and load balancer health checks.
5. **Device-agnostic JSON-RPC WebSocket** -- `/api/v1/ws/nio` accepts JSON-RPC 2.0 messages where the device UDID is a parameter, not part of the URL. This enables a single WebSocket connection to control multiple devices, which is ideal for CI/CD scripts that orchestrate test flows across a device farm.

## Challenges

1. **OpenAPI spec generation without macros** -- Generated the OpenAPI 3.0 spec programmatically in `openapi.rs` using nested struct construction. This is verbose (~900 lines) but avoids proc-macro dependencies (utoipa, paperclip). The trade-off is that the spec must be manually updated when endpoints change.
2. **Batch endpoint duplication** -- The `/api/v1/batch/tap`, `/api/v1/batch/swipe`, and `/api/v1/batch/input` endpoints share ~80% of their logic (device resolution, parallel execution, result aggregation). Each is a separate function with repeated patterns. A generic batch executor would reduce duplication.
3. **MetricsTracker Mutex contention** -- Screenshot latency samples use `Mutex<Vec<f64>>` which can contend under high load. The lock is held briefly (push + conditional remove), but a lock-free ring buffer would be more appropriate for a metrics path.
4. **WebSocket screenshot API connection lifecycle** -- The WS screenshot endpoint creates a new ATX client connection per WebSocket session. Should reuse the connection pool like the NIO handler does.

## Technical Debt

| Item | Severity | Description |
|------|----------|-------------|
| OpenAPI spec manually maintained | High | The 900-line `openapi.rs` must be manually updated when endpoints change. Consider utoipa or generating from route definitions. |
| Batch endpoint logic duplication | Medium | Three batch endpoints repeat device resolution and parallel execution logic. Extract a generic `execute_batch()` helper. |
| MetricsTracker uses Mutex | Low | `Mutex<Vec<f64>>` for latency samples. Replace with a lock-free circular buffer for the metrics hot path. |
| No API authentication | Medium | All API endpoints are unauthenticated. Fine for internal networks but needs API key or token auth for production exposure. |
| No rate limiting | Medium | API endpoints have no rate limiting. A single client could overwhelm the server with screenshot requests. |

## Action Items

1. Extract a generic `execute_batch()` function to deduplicate batch endpoint logic
2. Evaluate utoipa for automatic OpenAPI spec generation from route definitions
3. Implement API key authentication for the `/api/v1/` namespace
4. Add per-client rate limiting, especially for screenshot and batch endpoints
5. Replace MetricsTracker Mutex with a lock-free ring buffer for latency tracking

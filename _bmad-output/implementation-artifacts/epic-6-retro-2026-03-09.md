## Architecture Evolution

Each story added a clean layer without modifying layers below:

```
6-1: Foundation   →  ScrcpyManager + REST lifecycle (start/stop/list)
6-2: Control      →  Touch/key/swipe via scrcpy control socket
6-3: Distribution →  broadcast::Sender<BroadcastFrame> for 1:N frame relay
6-4: Persistence  →  Recording as another broadcast consumer → H.264 file
```

## Successes

1. **Layered architecture** — Zero rework across all 4 stories. Each layer built cleanly on the previous.
2. **Previous Story Intelligence** — Explicit documentation of lessons from prior stories reduced debug issues from 3 to 0 and code review findings from 14 to 0.
3. **Broadcast consumer composability** — Adding recording (6-4) required zero changes to producer or WS viewer code from 6-3.
4. **Zero new dependencies** — All needed crates already in Cargo.toml.
5. **Forward-design** — Story 6-1's `Arc<Mutex<ScrcpySession>>` in `ScrcpySessionEntry` was designed for 6-2 and 6-3 before those stories started.

## Challenges

1. **Rust async ownership** — Async closure lifetime constraints in 6-2 forced design pivot from closure-based `with_session` to returning `Arc<Mutex<>>` handles.
2. **TOCTOU race conditions** — Appeared in 3 of 4 stories. Solved with `starting` sentinel (6-1) and `get_session_with_info()` atomic retrieval (6-2).
3. **BufWriter flush on abort** — Code review in 6-4 caught that `task.abort()` prevents `BufWriter::flush()`, which would silently truncate recordings. Fixed with oneshot channel for graceful shutdown.
4. **Atomic frame counting** — `frame_count` tracked locally in consumer task was always 0 on explicit stop. Fixed with `Arc<AtomicU64>`.

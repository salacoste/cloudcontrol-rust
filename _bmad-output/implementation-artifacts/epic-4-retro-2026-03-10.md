## Architecture Evolution

Stories built in layers from multi-selection to full record/playback:

```
4-1: Multi-Selection UI   ->  BatchTapRequest/BatchSwipeRequest/BatchInputRequest with udids[] array
4-2: Synchronized Batch   ->  tokio::join_all for parallel execution + per-device Result tracking
4-3: Action Recording     ->  RecordingService + RecordingState (RwLock<HashMap>) + SQLite storage
4-4: Session Management   ->  pause/resume/cancel lifecycle + active/paused state machine
4-5: Recording Playback   ->  PlaybackSession state machine + background execution task
4-6: Batch Report Export  ->  BatchReport model + JSON/CSV/HTML export formats
```

## Successes

1. **Parallel batch execution with individual error isolation** -- `tokio::join_all` executes operations across all selected devices concurrently. Each device gets its own `Result`, so a failure on device A does not affect device B. The `BatchResponse` reports per-device success/failure with error codes.
2. **Recording state machine correctness** -- `RecordingState` uses `Arc<RwLock<RecordingStateInner>>` with explicit state transitions: active -> paused -> active -> stopped. The `should_capture()` method checks both `is_recording` and `!is_paused` atomically, preventing actions from being recorded during pause.
3. **Playback with pause/resume support** -- The background playback task polls `get_playback_session()` in a loop before each action. When paused, it sleeps 100ms and re-checks. This enables real-time pause/resume without losing position in the action sequence.
4. **ATX-first playback with ADB fallback** -- `execute_action_on_device()` mirrors the Epic 3 dual-path pattern: try ATX client first, fall back to ADB if ATX fails. Each action type (tap, swipe, input, keyevent) follows this pattern independently.
5. **Multi-format report export** -- Batch reports can be exported as JSON (API consumption), CSV (spreadsheet analysis), or styled HTML (human review). The HTML report uses dark theme styling with color-coded status indicators, generated entirely server-side without external templates.

## Challenges

1. **Recording vs playback state isolation** -- A device can be recording on one session while playing back another recording. `RecordingStateInner` uses separate `active_recordings` and `playback_sessions` HashMaps keyed by `device_udid` to prevent interference. A device cannot have two active recordings but can record while playing.
2. **Action sequence renumbering on delete** -- Deleting an action from mid-sequence requires renumbering all subsequent actions: `UPDATE SET sequence_order = sequence_order - 1 WHERE sequence_order > deleted_order`. This is done in a single SQL statement but could be slow for recordings with 1000+ actions.
3. **Playback speed implementation** -- Speed multiplier affects inter-action delay, not action execution time. At 2x speed, the 500ms base delay becomes 250ms. At 0.5x speed, it becomes 1000ms. This is a simplification -- true speed control would need to scale swipe durations too.
4. **MAX_BATCH_SIZE enforcement** -- Batch operations enforce a 20-device limit (`MAX_BATCH_SIZE`) to prevent resource exhaustion. The limit exists in both `api_v1.rs` and `openapi.rs` constants but could diverge if one is updated without the other.

| Pattern | Story | Reused In |

| Per-device error isolation in batch | 4-2 | Epic 5 (batch API endpoints) |
| Background task with polling state check | 4-5 | Epic 6 (recording consumer) |

| Multi-format export (JSON/CSV/HTML) | 4-6 | Report infrastructure |

## Technical Debt

| Item | Severity | Description |
|------|----------|-------------|
| MAX_BATCH_SIZE defined in two places | Medium | `api_v1.rs` and `openapi.rs` both define `MAX_BATCH_SIZE = 20`. Should be a single shared constant. |
| Playback does not scale action durations | Medium | Speed multiplier only affects inter-action delay, not swipe/long-press durations. 2x playback of a 2-second swipe still takes 2 seconds. |
| Recording storage unbounded | Low | No limit on number of recordings or actions per recording. Very long test sessions could produce thousands of actions in SQLite. |
| No recording export/import | Low | Recordings exist only in the local SQLite database. No JSON export for sharing between servers. |

## Action Items

1. Consolidate `MAX_BATCH_SIZE` into a single shared constant module
2. Implement duration scaling for swipe actions during playback speed adjustment
3. Add recording export to JSON format for portability between CloudControl instances
4. Consider adding a configurable retention policy for old recordings

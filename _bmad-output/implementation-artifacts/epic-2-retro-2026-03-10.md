## Architecture Evolution

Each story built on the previous screenshot capability:

```
2-1: Single Screenshot   ->  AtxClient.screenshot() + DeviceService.screenshot_base64()
2-2: WebSocket Streaming ->  NIO WebSocket handler with subscribe/unsubscribe protocol
2-3: Quality Config      ->  AtxClient.screenshot_scaled(scale, quality) — device-side processing
2-4: Multi-Device Batch  ->  ScreenshotCache with request deduplication via watch::channel
2-5: Resizing            ->  DeviceService.resize_jpeg() with image crate Nearest filter
```

## Successes

1. **Device-side screenshot processing** -- `screenshot_scaled(scale, quality)` delegates resize and JPEG compression to the u2 server on the device itself via `takeScreenshot(scale, quality)`. This eliminates the server-side decode-resize-encode cycle, reducing latency from ~800ms to ~200ms per frame.
2. **Dual capture path** -- WiFi devices use JSON-RPC `takeScreenshot` (fast, base64 response). USB devices fall back to `adb exec-out screencap -p` (slower, raw PNG). The fallback chain is transparent to callers.
3. **WebSocket binary frames** -- NIO streaming sends raw JPEG bytes as binary WebSocket frames instead of base64-encoded text messages. This reduces bandwidth by ~33% and eliminates client-side base64 decoding overhead.
4. **Request deduplication** -- `ScreenshotCache` uses `tokio::sync::watch` channels to coalesce concurrent screenshot requests for the same device. If 5 clients request a screenshot simultaneously, only one actual device call is made; the other 4 subscribe to the result via `try_subscribe()`.
5. **Performance telemetry** -- Screenshot streaming logs every 20th frame with timing breakdown: capture time, WS send time, total time, JPEG size in KB, and rolling average FPS. This data directly informed the quality/scale tuning decisions.

## Challenges

1. **Base64 line breaks from u2** -- The u2 server returns base64 with MIME-style `\n` line breaks. Required explicit `b64_data.replace('\n', "").replace('\r', "")` before decoding. This was a silent corruption bug that produced garbled images.
2. **Image crate performance** -- Default `image::resize` uses Lanczos3 filter which is 10x slower than Nearest. Switching to `FilterType::Nearest` (matching Python's `resample=0`) reduced server-side resize from ~120ms to ~12ms. Quality difference is imperceptible at 0.5x scale.
3. **Smart interval scheduling** -- Naive `sleep(interval)` between frames means actual frame period = capture_time + interval. Switched to measuring elapsed time and sleeping only the remaining interval: `if elapsed < min_interval { sleep(min_interval - elapsed) }`.
4. **Mock screenshot for stress testing** -- Added `OnceLock`-backed mock screenshot generator that creates a minimal 1080x2400 JPEG once and reuses it. Enables load testing the WebSocket infrastructure without real devices.

## Technical Debt

| Item | Severity | Description |
|------|----------|-------------|
| Screenshot cache eviction is FIFO not LRU | Low | `ScreenshotCache` evicts the first entry from DashMap iterator (arbitrary order) when full. Should use moka or track access time. |
| No backpressure on WebSocket stream | Medium | If the client can't consume frames fast enough, the server keeps sending. Should detect slow consumers and skip frames. |
| Hardcoded scale=0.5 quality=40 in NIO | Low | NIO streaming defaults are hardcoded. Should read from client subscribe message or server config. |
| No PNG output path | Low | All screenshots convert to JPEG. PNG output would require skipping the JPEG encode step. |

## Action Items

1. Add WebSocket backpressure detection to skip frames for slow consumers
2. Replace ScreenshotCache FIFO eviction with LRU or timestamp-based eviction
3. Make NIO streaming scale/quality configurable per-client via subscribe message parameters

## Successes

1. **Middleware architecture compounds** — Story 12.1 established the Transform + Service pattern. Story 12.2 reused it in half the implementation time. The pattern is now a documented template for future middleware.

2. **Code review as critical quality gate** — Every story had review fixes. Story 12.2 had dead code and ordering bugs. Story 12.4 had missing validation bounds. Story 12.6 had incomplete API coverage. The adversarial review process consistently caught issues before merge.

3. **Zero production incidents** — Major security changes (auth, rate limiting), crash protection, and graceful shutdown all deployed without breaking existing functionality.

4. **Backward compatibility preserved** — All new features are opt-in. No API key configured = auth disabled. No rate limit config = rate limiting disabled. Existing deployments work unchanged.

5. **Performance optimization** — VecDeque for MetricsTracker (O(n) → O(1)). spawn_blocking for image processing (no async runtime blocking). Connection pool and cache sizes now configurable.

6. **Test investment** — 180 new tests across 6 stories. Every middleware has unit tests. Every config field has validation tests. Test-to-code ratio is healthy.

## Challenges

1. **Self-verification gaps** — Story 12.6 marked done but missed api_v1.rs endpoints (get_screenshot, ws_screenshot). Code review found the gaps. Pattern: task completion claims sometimes ahead of actual implementation.

2. **Actix-web middleware complexity** — Story 12.2 rate limit headers on successful responses blocked by Either<Map<Fut>, Ready> return type. Would require Pin<Box<dyn Future>> pattern. Marked as N/A with documented limitation.

3. **Rust async ownership puzzles** — Story 12.6 ws_screenshot required cloning state for inner spawned task while keeping reference for cleanup. Classic Rust async pattern requiring careful planning.

4. **AC coverage verification** — Stories sometimes complete with all tasks checked but missing coverage of some ACs in certain files. The code review is the safety net, but self-verification could be stronger.
